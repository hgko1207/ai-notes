# Gemma 4 MTP + 은폐 모델 논란 — 구글이 너무 강한 오픈소스를 숨겼나?

> 한 줄 요약: 구글이 Gemma 4 MTP(멀티토큰 예측)로 추론 속도를 3배 빠르게 만들었지만, 원래 배포판에서 일부러 제거했다가 커뮤니티 리버스 엔지니어링으로 들통난 후 공개됐다는 논란이 일고 있습니다.

## 핵심 요약
- **MTP(Multi-Token Prediction)**: 여러 토큰을 한 번에 예측해 추론 속도를 **최대 3배** 향상
- **품질 저하 없음**: 추론 논리·품질 저하 없이 속도만 향상
- **은폐 의혹**: Gemma 4 MTP가 원래 있었으나 배포판에서 제거된 후 커뮤니티가 리버스 엔지니어링으로 발견
- **122B 모델 삭제 의혹**: 122 빌리언 파라미터 모델도 유출 후 삭제된 흔적 발견
- **상용 API 보호 추정**: 너무 강력한 오픈소스가 자사 유료 모델(Flash Lite 등) 시장을 위협한다는 분석

## 본문

### MTP — 멀티토큰 예측의 마법

#### 어떤 기술인가

기존 LLM은 한 번에 **하나의 토큰**만 예측하는 방식입니다.

```
입력 → 모델 → 토큰 1 → 모델 → 토큰 2 → 모델 → 토큰 3 → ...
```

**MTP(Multi-Token Prediction)**는 한 번에 **여러 토큰**을 동시 예측합니다.

```
입력 → 모델 → [토큰 1, 2, 3, 4] 한 번에 → 검증 → 출력
```

#### 속도 향상 원리

| 요인 | 효과 |
|------|------|
| 모델 가중치 한 번 읽기 | 메모리 접근 횟수 감소 |
| 여러 토큰 동시 검증 | 병렬 처리로 속도 향상 |
| 토큰 재사용 가능 | 계산 효율 극대화 |

**결과**: 최대 **3배 속도 향상**, 품질 저하 없음

### Gemma 4 MTP의 성능

| 항목 | 기존 Gemma 4 | Gemma 4 MTP |
|------|------------|------------|
| 추론 속도 | 기준 | **최대 3배** |
| 품질 | 동일 | 동일 |
| 추론 논리 | 동일 | 동일 |

특히 **온디바이스(스마트폰, 노트북)** 환경에서 큰 성능 향상이 있어 로컬 AI 사용자에게 매우 유용합니다.

### 은폐 논란 — 들통난 후의 공개?

#### 커뮤니티의 발견

> "원래 Gemma 4 MTP가 은폐 상태였다고 합니다. 커뮤니티가 파헤쳐서 뒤늦게 공개됐다."

#### 구체적 정황

| 단계 | 내용 |
|------|------|
| 1 | 구글이 MTP로 학습시킨 Gemma 4 모델을 만듦 |
| 2 | 배포판에서 MTP 기능을 **의도적으로 제거** |
| 3 | 커뮤니티 개발자들이 **리버스 엔지니어링** |
| 4 | MTP 기능이 모델 내부에 남아있다는 사실 발견 |
| 5 | 들통난 후 구글이 공식 공개 |

#### 왜 숨겼을까

가장 유력한 분석은 다음과 같습니다.

> "상용 API의 경쟁력을 위협하기 때문에 의도적으로 너프(약화)했다."

만약 무료 오픈소스 모델이 너무 빠르고 품질도 좋다면, 다음 같은 상황이 발생할 수 있습니다.

- 사용자들이 유료 **Gemini 3.1 Flash Lite** 같은 가벼운 모델을 안 씀
- **API 매출 감소** 우려
- 자사 클라우드 매출 위협

---

### 122B 모델 삭제 의혹 — 더 충격적인 사례

#### 발견된 흔적

> "유출로 삭제된 122 빌리언 모델도 있었다고 합니다."

122 빌리언 파라미터 모델은 어마어마한 성능을 보였을 가능성이 큽니다. 이런 모델이 무료 오픈소스로 풀린다면:

| 영향 | 내용 |
|------|------|
| Gemini Pro 같은 유료 모델 매출 감소 | 큰 타격 |
| 클라우드 컴퓨팅 매출 감소 | 자체 GPU로 돌려쓰면 됨 |
| 경쟁사 대비 우위 약화 | 무료 공개 = 경쟁사도 활용 가능 |

#### 두 가지 추측

1. **상용 API 보호용**: 너무 강력한 모델은 유료 시장 위협
2. **Google I/O를 위한 아껴두기**: 다음 달 발표를 위한 비축 (긍정적 해석)

### 시사점 — 오픈소스 vs 상용 API의 딜레마

#### 빅테크의 고민

| 옵션 | 장점 | 단점 |
|------|------|------|
| 강력한 오픈소스 공개 | 개발자 생태계 확보, 평판 향상 | 자사 유료 모델 매출 감소 |
| 약하게 너프해서 공개 | 매출 보호 | **들통나면 신뢰 손실** |
| 비공개 유지 | 매출 보호 | 폐쇄적 이미지, 경쟁사에 밀림 |

이번 사태는 빅테크들이 이 딜레마 사이에서 어떻게 균형을 잡으려 하는지 보여줍니다.

#### 커뮤니티의 힘

리버스 엔지니어링을 통해 숨겨진 기능을 발견하고 우회 지원까지 만든 커뮤니티는 **AI 오픈소스 생태계의 자정 능력**을 보여줬습니다.
